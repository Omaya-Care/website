## 0. Governing constraints (these override the raw scan)

Every recommendation below is filtered through three non-negotiable rules from the project owner:

1. **Pixel-identical output.** The UI must not change *at all*. No component moved, shifted, reordered, split, or restructured in any way that alters rendered layout or appearance. Breakpoints, Tailwind variants, and fractional `rem`/`px` dimensions are the visual contract and are **out of scope to touch**.
2. **Maximum performance + responsiveness.** This is the actual goal — delivered through *behavior and delivery* changes (network, paint scheduling, caching), never through layout changes.
3. **Best-practice, reasoned, per-section.** Each change carries its rationale and a UI-safety verdict. Nothing is applied blindly to satisfy a linter score.

> **Consequence:** the raw react-doctor score (45/100) is **not the target metric.** A large share of the findings are false positives from the scanner mis-modelling this repo (see §1). We optimize for *real* performance + correctness under the pixel-identical mandate, not for the score.

### Verdict legend

| Verdict | Meaning |
|---|---|
| ✅ **SAFE** | Zero visual risk. Attribute/behavior/config only. Do it. |
| 🔍 **VERIFY-FIRST** | Safe only after a named check (grep, build-graph, or visual diff) passes. |
| ⛔ **SKIP / REJECT** | Changes output, moves components, or is a false positive. Do not do under the constraint. |

## 1. Critical reframe — react-doctor mis-scored this repo

The scanner assumed a single-entry SPA. **This is a multi-page app with 6 HTML entry points**, each with its own `main.tsx` (`vite.config.ts` → `rollupOptions.input`):

| HTML entry | Mounts | Route |
|---|---|---|
| `index.html` | `src/main.tsx` | `/` |
| `contact.html` | `src/pages/contact/main.tsx` | `/contact` |
| `privacy.html` | `src/pages/privacy/main.tsx` | `/privacy` |
| `faq.html` | `src/pages/faq/main.tsx` | `/faq` |
| `tos.html` **and** `terms.html` | `src/pages/tos/main.tsx` | `/tos`, `/terms` |
| `preview.html` | — (not in build `input`) | static asset gallery, unshipped |

Because react-doctor walked only from a single root, it produced **two false-positive floods** that dominate the score:

- **`unused-file` (77)** — the FAQ / Privacy / TOS / Contact page trees (`ditto/`, `sections/`, `content.ts`, `lib/utils.ts`, etc.) are **reachable and rendered** via their own entries. They are *not* dead code. Verified live: e.g. `src/pages/privacy/page.tsx:1` imports `ditto/DittoMotion`, rendered at line 23.
- **`effect-needs-cleanup` (5 "errors")** — the flagged `useEffect`s **already return full cleanup functions**. Confirmed in source: `src/ditto/DittoMotion.tsx:189–195` clears every interval, disconnects the `IntersectionObserver`, and clears all timers; `FaqAccordion.tsx` pairs each `addEventListener` with a `removeEventListener` in its returned cleanup; `DropdownMenu.tsx` uses an `AbortController` (`ac.abort()`). react-doctor can't follow the `intervals.push(setInterval(...))` → drain-in-cleanup indirection.

> **The original plan's #1 action — "delete 77 unused files, biggest lever, lowest risk" — is inverted. Under the constraint it is the single highest-risk action.** Deleting those files breaks the build and removes rendered pages. **Do not delete the page trees.** No router work is needed either — the MPA already wires them.

**Recommended first step:** re-run react-doctor MPA-aware (point it at all 6 `main.tsx` entries if supported) so the score reflects reality before any work.

### Detail on the actionable items

**#4 `jsx-no-target-blank` — ✅ SAFE.** `src/pages/{privacy,tos}/sections/gallery-showcase-section.tsx:163` already has `rel="nofollow"`, just missing `noreferrer`:
```tsx
rel="nofollow noreferrer"   // closes reverse-tabnabbing + strips Referer; zero paint effect
```

**#6 `button-has-type` — ✅ SAFE.** `src/sections/hero-section.tsx:9` and `src/pages/faq/sections/hero-section.tsx:9` are the modal **Close** buttons (not submits):
```tsx
<button type="button" aria-label="Close modal" id="sp2Close" className="…">
```
A bare `<button>` defaults to `type="submit"`; `type` has no rendered appearance.

**#7 `no-array-index-as-key` — ⛔ SKIP (all 35).** Every hit maps a frozen module-scope/prop array that never reorders or filters (e.g. `contact/page.tsx:312,327` fixed 4-item carousel; `page.tsx:243` a *tripled* infinite-loop track where a stable id would actually **collide**). For a static list, `key={i}` is semantically identical. Swapping to `key={d.id}` forces a one-time unmount+remount of every item → a possible **one-frame flash / replayed entrance animation** on the reveal-animated tiles = a visual diff for zero correctness gain. If the score must be cleared, prefer a per-line suppression over editing keys.

**#8 `rendering-svg-precision` — ⛔ SKIP.** Inline `<path d="…">` with sub-pixel coords paired with matching high-precision `origin-[7.32031px]` transforms. Rounding path data while transforms stay precise → hairline misalignment at some zoom/DPR. Byte savings are trivial and already recovered losslessly by **Brotli** (see §4-H).

**#2 `unused-file` — 🔍 delete only proven orphans.** After the MPA-aware re-scan, a *handful* may be genuine orphans not in any page tree (candidates flagged by the reviewers: `src/sections/sticky-dual-section.tsx` — 0 importers; possibly `gallery-showcase-section2.tsx`, stray `svg-icon*`/`text-link*` variants). Delete each **only** after: (a) repo-wide reverse-import grep returns nothing across *all* entries, and (b) `vite build` still succeeds with an unchanged emitted module graph.

## 3. Verification strategy — how we PROVE zero UI change

Playwright is already a dependency. Use it as the hard gate, not just for tests.

1. **Baseline (before any change):** `vite build && vite preview`, then a Playwright script that visits all **6 routes** (`/`, `/contact`, `/privacy`, `/faq`, `/tos`, `/terms`) at **3 viewports** (1440 / 768 / 390) and captures `fullPage` screenshots into `__visual_baseline__/`. Also capture the **interactive states** static shots miss: FAQ accordion expanded, privacy/tos `DropdownMenu` open, a `DittoMotion` reveal frame (freeze the timer for determinism).
2. **Per-batch gate:** after each batch, re-run with `expect(page).toHaveScreenshot({ maxDiffPixels: 0 })`. **Any non-zero diff blocks the batch.**
3. **Build-graph diff (deletions only):** diff the emitted chunk/module list before/after; a deletion is valid only if the build still succeeds and no other chunk lost a needed module.
4. **DOM-structure diff (belt-and-suspenders):** for any key/attribute change, snapshot `outerHTML` per route and diff — must be byte-identical.
5. **Determinism:** disable animations / `prefers-reduced-motion` for screenshots; assert on a fixed frame for timer components so diffs aren't flaky.

## 4. The real work — performance wins the scan never looked at

These are where the actual perf/responsiveness goal is met. **All are attribute- or config-level: computed layout stays byte-identical at every breakpoint.** Ranked by payoff.

| # | Win | Verdict | Detail |
|---|---|---|---|
| A | **Image `loading="lazy"` + `decoding="async"`** | ✅ SAFE | Below-the-fold `.avif` (heavy site). Boxes already sized by Tailwind → **no CLS**. **Do NOT lazy-load the LCP/hero image** — give it `loading="eager"` + `fetchpriority="high"`. Verify which image is LCP first. |
| B | **Vendor `manualChunks` split** | ✅ SAFE | No `output.manualChunks` today → React/ReactDOM re-bundle per entry; `/`→`/faq` re-downloads React. Isolate `react`+`react-dom` into a long-cache vendor chunk shared across all 6 entries. **Highest-value item.** Byte-identical render. |
| C | **Preload LCP image + primary font** | ✅ SAFE | `<link rel="preload" as="image" fetchpriority="high">` + `<link rel="preload" as="font" crossorigin>` in each entry `<head>`. Earlier critical fetch → better LCP. Verify the preloaded image is actually the LCP one. |
| D | **`preconnect` / `dns-prefetch`** | ✅ SAFE | For third-party origins (LegitScript link, any analytics/widget host). Pure network hint. |
| E | **Passive event listeners** | ✅ SAFE | `DropdownMenu.tsx:62` already uses `{ passive: true }` — good. Audit other `scroll`/`touch`/`wheel` listeners; add `{ passive: true }` where no `preventDefault`. Smoother scroll, no visual change. |
| F | **Brotli/gzip + modern `build.target`** | ✅ SAFE | Ensure host serves Brotli (recovers all "SVG precision" bytes and far more, losslessly); set `build.target` to actual supported browsers to drop legacy transpile. Confirm `cssCodeSplit` on so each entry ships only its CSS. |
| G | **`font-display`** | 🔍 VERIFY-FIRST | `swap` risks a FOUT transient (a visual change); `optional` risks a permanent fallback. **Safest: self-host + preload (C) so the font is ready before first paint and never swaps.** Don't change `font-display` blindly. |
| H | **Memoization** | ⛔ SKIP unless measured | Premature on a static marketing site; wrong `useMemo`/`useCallback` over `slide`/`index` could break a carousel. Only with a profiler-proven hotspot. |

> **Responsiveness without touching breakpoints:** every win above is delivery/paint-scheduling, not CSS. Do **not** edit any `max-md:`/`max-lg:`/`2xl:` variant or fractional dimension — those *are* the pixel-identical contract. Before adding `loading="lazy"`, confirm each below-fold `<img>` has a sized box (Tailwind classes already provide this) so lazy-loading introduces zero CLS.
